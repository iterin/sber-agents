# Workflow разработки по tasklist.md (KISS)

Цель: дисциплинированно реализовать бота из `docs/vision.md` по шагам из `docs/tasklist.md`.

## Принципы
- Делать только то, что в текущей итерации (KISS, YAGNI, MVP).
- Каждый шаг проверяем тестом (минимальный сценарий).
- Не двигаемся дальше без подтверждения.

## Роли
- Кодовый ассистент: предлагает решение, реализует, тестирует, фиксирует прогресс.
- Ответственный: согласует решение, подтверждает результат и переход к следующей итерации.

## Цикл каждой итерации
1) Согласование решения (до кода)
- Кратко описать подход и показать ключевые отрезки кода/псевдокода, которые будут добавлены/изменены.
- Привязать к задачам текущей итерации из `docs/tasklist.md`.
- Дождаться одобрения.

2) Реализация
- Внести согласованные изменения минимальным объёмом.
- Соблюдать читаемость, конфиденциальность и безопасность.

3) Верификация
- Выполнить минимальные тесты, указанные в итерации.
- Зафиксировать результат (успешно/ошибка, краткая заметка).

4) Подтверждение
- Показать важные фрагменты кода и результаты тестов.
- Дождаться подтверждения от ответственного.

5) Обновление прогресса
- Обновить таблицу прогресса сверху в `docs/tasklist.md` (статус, проценты, примечания).
- Отметить выполненные чекбоксы задач текущей итерации.

6) Коммит
- Сделать коммит с информативным сообщением: `iter-<n>: <кратко что сделано>`.

7) Переход к следующей итерации
- Запросить согласование перехода.
- После одобрения — повторить цикл для следующей итерации.

## Запреты
- Не добавлять функционал вне текущей итерации.
- Не менять план без согласования.
- Не пропускать согласование, подтверждение и обновление тасклиста.
